# Phase 2 Design Decisions

Implementation decisions made during Phase 2 ("The Editor Core") that were NOT explicitly specified in the guidance docs (AGENTS.md, FRONTEND.md, BACKEND.md, DATABASE.md, TESTING.md).

## Backend Architecture

### 1. Plain Express Router instead of tsoa for routes

tsoa is listed as a dependency and is the recommended API framework in BACKEND.md, but Phase 2 routes use plain Express Router factories. Rationale: faster iteration for the PoC branch. tsoa decorators and OpenAPI generation can be added in a later pass without changing the service/repository layers.

### 2. Manual DI wiring in app.ts (no DI container)

Dependencies are manually instantiated and wired in `app.ts`: `db -> repos -> services -> routes`. No DI container library (e.g., tsyringe, inversify) is used. This keeps the dependency graph explicit and easy to follow. A container can be introduced when the number of services grows.

### 3. Route factory pattern

Routes are exported as factory functions (`create_document_routes(service)`) that return an Express Router. This decouples route registration from service instantiation and makes the routes independently testable.

### 4. Auto-creating initial document version (v1)

When a document is created, an initial version (v1) is automatically inserted with empty ProseMirror content `{type: "doc", content: []}`. This ensures `get_document` always has content to return and avoids null-content edge cases in the editor.

### 5. Version number incrementing strategy

New versions are numbered by querying the latest version (`find_latest`) and incrementing by 1. This is simple and correct for single-user editing. For concurrent editing (Phase 4 collab), this strategy may need to be revisited with optimistic locking or a sequence.

### 6. Document routes at `/documents` not `/api/v1/documents`

The document-service mounts routes at `/documents` and `/workspaces`. The API Gateway's reverse proxy adds the `/api/v1` prefix when forwarding. This keeps each service unaware of the gateway's URL namespace.

### 7. Workspace routes on document-service (not separate service)

Workspaces are managed by the document-service rather than a dedicated workspace-service. They are tightly coupled to documents and do not warrant a separate deployment at this scale. Can be extracted later if needed.

### 8. `content_json` and `content_text` in `document_versions` not `documents`

Document content is stored exclusively in the `document_versions` table, not directly on `documents`. The `documents` table holds metadata only. This enables version history, undo, and future branching without data duplication.

### 9. Slug auto-generation from workspace name

When creating a workspace without an explicit slug, one is auto-generated by lowercasing the name and replacing non-alphanumeric characters with hyphens. This provides URL-friendly workspace identifiers without requiring user input.

## Frontend Architecture

### 10. `api_fetch` wrapper with automatic JWT attachment

A generic `api_fetch<T>()` function wraps the native `fetch` API, automatically attaching the Supabase session JWT as a Bearer token. This centralizes auth header management and error handling.

### 11. QueryClient defaults: staleTime 30s, retry 1

TanStack Query is configured with a 30-second stale time (balances freshness with network efficiency for document editing) and a single retry (avoids hammering the API on persistent failures).

### 12. Auto-creating default workspace on first document creation

The DocumentList page auto-creates a "My Workspace" workspace if the user has none, then immediately creates the document within it. This removes onboarding friction for the PoC.

### 13. First workspace as "active workspace" (no workspace switcher)

The UI uses `workspaces[0]` as the active workspace. A proper workspace switcher/selector is deferred to a later phase. This is a deliberate PoC simplification.

### 14. Debounced autosave at 1000ms

The EditorPage autosaves document content 1 second after the last keystroke. This balances responsiveness (user sees "Saved" quickly) with network efficiency (not saving on every keypress). The delay is a constant that can be made configurable.

### 15. Save status indicator pattern

The EditorPage uses a `SaveStatus` enum (`idle | saving | saved | error`) rendered as a small label in the toolbar. The "Saved" indicator auto-clears after 2 seconds. This provides user confidence without being intrusive.

### 16. Default empty ProseMirror content uses `{type: "paragraph"}` (no empty text nodes)

The initial editor content uses `{type: "doc", content: [{type: "paragraph"}]}` rather than including an empty text node. ProseMirror rejects empty text nodes (`""`), so this avoids runtime warnings.
